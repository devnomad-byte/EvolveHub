# Chat Typing Stream Design

- 日期：2026-04-18
- 执行者：Codex
- 范围：`frontend/src/components/apps/chat/ChatApp.vue` 及其直接依赖的前端显示层

## 1. 目标

将当前聊天页的“收到 `chunk` 就直接整段落屏”改为“跟随上游的打字式流式渲染”：

- 上游一旦返回 `chunk`，前端立即开始消费，不额外等待整轮结束。
- 单个 `chunk` 内部不直接整段显示，而是平滑拆字后逐步渲染。
- 当上游分片很粗时，视觉上仍保持稳定的“正在打字”效果。
- 不修改现有 SSE 协议，不改后端事件类型。

## 2. 当前现状

当前聊天页已经能消费 SSE：

- `chatApi.sendMessage()` 逐块读取 `text/event-stream`
- `ChatApp.vue` 在 `handleStreamEvent()` 中直接执行：
  - `assistantMessage.content += event.content || ''`

这意味着：

- 如果上游本身分片细，页面看起来像流式。
- 如果上游一次返回很大 `chunk`，页面会整段跳出，缺少稳定打字感。

## 3. 推荐方案

采用“时间片驱动队列”的前端显示层方案：

- 收到 `chunk` / `reasoning` 时，不直接写入 `assistantMessage.content`
- 先把文本放入当前消息对应的待渲染缓冲区
- 用一个单一调度器持续从缓冲区取字符，按节奏写入页面
- 缓冲越多，单帧吐字越快；缓冲变少，自动降速，保持自然观感

该方案不改变接口契约，只改变前端消息落屏方式。

## 4. 状态设计

为当前正在输出的 assistant 消息维护一个本地渲染状态：

- `pendingText`
  含义：尚未落屏的文本缓冲区
- `isStreaming`
  含义：当前消息是否仍在接收上游流
- `isFlushing`
  含义：当前是否正在执行打字调度
- `streamDone`
  含义：是否已经收到 `done`

状态约束：

- 只有当前轮 assistant 消息参与打字调度
- 新一轮发送时重置为新的渲染状态
- `done` 到来时，如果 `pendingText` 未清空，不能立刻视为本轮显示完成

## 5. 渲染算法

### 5.1 入队

当收到 `chunk` 或 `reasoning`：

- 将文本追加到 `pendingText`
- 如果调度器未运行，则立即启动调度器

### 5.2 调度

调度器使用短周期时间片推进，推荐 `requestAnimationFrame` 驱动：

- 每一帧根据 `pendingText.length` 决定本帧输出字符数
- 推荐规则：
  - `1-20` 字：每帧 `1` 字
  - `21-80` 字：每帧 `2-3` 字
  - `>80` 字：每帧 `4-6` 字

这样既不会在短回复里显得太快，也不会在长回复积压时落后太多。

### 5.3 完成

当满足以下条件时，才真正结束本轮打字：

- `streamDone === true`
- `pendingText === ''`

在此之前，消息仍视为“正在输出”。

## 6. 视觉表现

当前 assistant 消息增加一个轻量光标：

- 光标显示条件：
  - 正在接收流，或
  - 缓冲区还没刷完
- 光标隐藏条件：
  - 已收到 `done`
  - 且缓冲区已清空

光标只作用于当前输出中的 assistant 消息，不影响历史消息。

## 7. 非文本事件处理

以下事件不走打字缓冲，保持即时落屏：

- `tool_call`
- `tool_result`
- `memory_warning`
- `error`
- `session_created`

原因：

- 它们属于状态事件或工具事件，延迟显示会影响可理解性和排障效率。

## 8. 边界条件

### 8.1 上游单次返回超大 chunk

表现：
- 单次入队文本很多

处理：
- 动态提高每帧吐字量
- 保证视觉上仍连续输出，但不明显滞后

### 8.2 提前收到 done

表现：
- 上游流结束，但 `pendingText` 还没刷完

处理：
- 等待缓冲清空后，再结束当前消息流式态

### 8.3 用户切换会话

处理：
- 终止当前页面正在运行的打字调度器
- 清理当前会话对应的流式渲染状态
- 避免旧会话缓冲区继续写入新会话视图

### 8.4 连续快速发消息

处理：
- `sending` 仍沿用现有互斥逻辑
- 同一时间只允许一个 assistant 消息处于活动流式态

### 8.5 错误中断

处理：
- 若收到 `error`，直接把错误消息写入当前 assistant 消息
- 清空 `pendingText`
- 停止光标与调度器

## 9. 实现边界

本次实现仅修改前端显示层：

- `frontend/src/components/apps/chat/ChatApp.vue`

可选的小型类型补充：

- 若需要提炼渲染状态类型，可放入 `frontend/src/types/index.ts`

不改动：

- 后端 SSE 协议
- `chatApi.sendMessage()` 的事件读取协议
- 消息持久化结构
- 工具调用事件格式

## 10. 测试与验证

### 自动验证

- 前端类型检查：`npm run build`
  说明：当前仓库存在其他模块的既有 TS 报错，本次只确保不新增聊天流式相关错误

### 手工验证

1. 发送短回复，确认逐字显示且无明显滞后
2. 发送长回复，确认持续打字且不会整段突然跳出
3. 人为制造粗粒度 chunk，确认单 chunk 也能平滑拆字显示
4. 发送包含 `tool_call/tool_result` 的问题，确认工具事件仍即时出现
5. 在回复中切换会话，确认不会串写到其他会话

## 11. 风险

- 如果每帧吐字过慢，长回复会积压，视觉上落后模型输出
- 如果每帧吐字过快，打字感会丢失，退化成“快速整段刷出”
- 会话切换和错误中断如果清理不彻底，容易出现串流

## 12. 结论

本方案以最小改动实现“稳定的打字式流式渲染”：

- 不动后端协议
- 不动消息结构
- 只增强前端 `chunk` 的显示层

推荐按此方案进入实现。
